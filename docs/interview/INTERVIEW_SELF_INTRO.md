# 自我介绍 (3 种长度) — ERP AI Copilot 项目

> 适用: AI 应用工程师 / Python 后端 / LLM 应用开发 / Agent 开发岗位
> 风格: 口语化, 不背稿. 三种长度按"面试官给的时间"切换.
> 用法: 面试前 1 天对镜子读, 录音回放检查口头禅 (呃/然后/就是).

---

## 一、1 分钟版 (150 字, 简短自我介绍)

### 适用场景
- 群面 / 1 对 N 面试 (5 个候选人轮流)
- 面试官明确说"先简单介绍一下"
- 视频面试开头 60 秒

### 节奏
- 0-10 秒: 姓名 + 来自哪里 + 面试岗位
- 10-30 秒: 一句话项目定位
- 30-50 秒: 1 个最亮的技术数字
- 50-60 秒: 过渡到项目演示

### 重点
**只讲 1 件事**: 项目是 ERP AI Copilot, 不是"多 Agent 框架".

### 完整口述稿 (直接读)

> 您好, 我叫 XXX, 来自深圳, 今天面试 AI 应用工程师岗位. · 我做的项目叫 **ERP AI Copilot** — 面向 ERP 业务场景的 AI 助手, 用 4 个 Agent 协作回答库存、采购、销售、制度问答. · 项目用 15 天做完, 5 张表真实数据 + 18 篇中文知识库 + 168 个测试全过. · 如果您感兴趣, 我可以用 2 分钟过一下技术亮点, 或者直接演示.

### 字数核对
正文 78 字, 加项目名 90 字, 加过渡 110 字. 1 分钟讲完, 留 5-10 秒给面试官反应.

### 过渡到项目演示
"我可以打开 GitHub 仓库带您看代码, 或者直接跑一个单 Agent 演示."

### 兜底 (如果卡壳)
"我做的是 ERP AI Copilot, 15 天完成, 168 测试全过, 5 张表 + 18 篇 KB + 4 Agent 协作."

## 二、3 分钟版 (400 字, 项目 + 1 个技术亮点)

### 适用场景
- 1 对 1 技术面, 面试官说"介绍一下你自己和项目"
- 二面 / 终面开场
- 视频面试前 3 分钟

### 节奏
- 0-15 秒: 姓名 + 背景 (Java 1-3 年, 转 AI)
- 15-45 秒: 项目背景 (MACS → ERP AI Copilot 的定位转变)
- 45-120 秒: 项目做了什么 (4 块架构, 1 句话 1 块)
- 120-180 秒: 1 个最亮的技术亮点 (推荐 NL→SQL 4 层防护)
- 180-200 秒: 1 个超出 Plan 的项 (推荐 168 测试金字塔)
- 200-220 秒: 过渡到项目演示 / 提问

### 重点
**项目 4 块 + 1 个亮点 + 1 个 Plan 外项**. 亮点选 NL→SQL 4 层防护是因为它最具体、最安全向, 面试官容易接话.

### 完整口述稿 (直接读)

> 您好, 我叫 XXX, 来自深圳, 有 1-3 年 Java 后端经验, 最近半年转 AI 应用方向, 今天面试 AI 应用工程师. · 我做的项目叫 **ERP AI Copilot** — 面向企业 ERP 系统的 AI 助手, GitHub 公开 (`blank5this/MACS`).
>
> · 这个项目的起因是: 我之前做了一个通用的多 Agent 框架, 但投简历时发现面试官**看不出业务价值**. 所以我用 15 天时间, 把框架重做成一个**聚焦 ERP 业务场景的 AI 助手产品** — 从"框架"到"产品"的定位转变. · 这是我做完反思后认为**最有价值的一步**.
>
> · 项目分 4 块. **数据层** — PostgreSQL 5 张表 (产品/销售订单/采购订单/供应商/库存), 1000+ 行 Faker 种子数据, 模拟真实中型 ERP. **Agent 层** — 实际跑过单 Agent 7 工具混合, prompt 1200 token 会膨胀; 后来拆成多 Agent: Planner 拆任务, Inventory Analyst 查库存, Purchase Specialist 看采购, Report Writer 出报告, 4 级 Hierarchical 编排. **检索层** — 18 篇中文 ERP 制度文档, 用 char-ngram + BM25 + RRF 三路融合, 解决中文短查询召回不稳. **工程化** — 4 层 SQL 注入防护, FastAPI 4 endpoints 加 3 Tab Web UI, GitHub Actions 4 job CI, 168 个测试全过.
>
> · 我最想讲的一个技术亮点是 **NL→SQL 4 层防护**. 用户问中文问题, LLM 翻译成 SQL, 我做了 4 层: 第一层 AST 解析强制只允许 SELECT; 第二层关键字黑名单挡 `INSERT` / `DELETE` / `DROP` 等; 第三层表列白名单限定只能查 5 张业务表; 第四层 psycopg 参数化绑定. 这 4 层加起来, 即使用户输入"把 users 表删掉", 在 AST 阶段就被拒. 我配套写了 `tests/test_nl2sql_safety.py` 覆盖 4 层, 每个 case 单测.
>
> · Plan 没要求但我顺手做的: **168 个测试金字塔** — 单元 ~150 个 + 集成 ~23 个 + E2E ~6 个, 按模块分 12 个文件, CI 自动跑. 这块是工程质量的具体体现.
>
> · 我可以现场演示. 我有 3 段 60s 视频脚本, 一个跑单 Agent, 一个跑多 Agent 协作, 一个跑 RAG 检索. 您想先看哪个? 或者我直接打开 GitHub 带您过一下代码结构.

### 字数核对
正文 ~450 字, 3 分钟讲完. 关键停顿用 `·` 标.

### 过渡到项目演示
- 演示派: "我先跑一个 2 分钟的多 Agent 演示, 边跑边讲 Planner 怎么拆."
- 代码派: "我打开 GitHub, 先带您看 4 个 Agent template 在 `macs_pkg/erp/agents/templates.py`."
- 视频派: "我录了 3 段 60s 视频, 投影出来讲."

### 兜底 (如果时间被压缩到 1 分半)
跳过"4 块架构"那段, 直接讲"项目定位 + NL→SQL 4 层防护 + 168 测试 + 演示".

### 容易挂的 3 个点
1. **"4 块架构"念得太快** — 听众记不住. 解决: 每块说完停顿 1 秒, 用"第一/第二/第三/第四" 强化.
2. **讲技术亮点时忘记配数字** — "4 层防护" 后面不跟"挡 DROP 的具体例子"就单薄. 解决: 背"把 users 表删掉"这个 case.
3. **过渡到演示时太突然** — "我可以演示" 之后直接跳, 面试官没准备. 解决: "您想先看哪个? 或者我直接打开 GitHub" 给选项.

## 三、5 分钟版 (700 字, + 1 个失败案例 + 反思)

### 适用场景
- 终面 / 总监面 / 技术 Leader 面
- 面试官明确说"讲讲你做过最有挑战的项目"
- 公司给了 30 分钟面试, 开场要 5-7 分钟

### 节奏
- 0-20 秒: 姓名 + 背景 + 面试岗位
- 20-60 秒: 项目背景 + 定位转变 (MACS → ERP AI Copilot)
- 60-180 秒: 项目做了什么 (4 块架构, 重点讲 Agent 层和检索层)
- 180-240 秒: 1 个最亮的技术亮点 (NL→SQL 4 层防护)
- 240-300 秒: 1 个失败案例 + 反思 (推荐 v1.0.0 RuntimeEngine 吞异常)
- 300-360 秒: 1 个超出 Plan 项 + 数据 (168 测试金字塔 + 数字)
- 360-400 秒: 1 个反问面试官 (留白: 问团队 AI 落地阶段)

### 重点
**5 分钟版的核心是讲"我犯过错, 我反思了, 我改了"**. 失败案例是亮点, 不是减分项. 面试官通过失败案例判断候选人的工程成熟度.

## 五、3 个分支应对 (面试官中途可能问)

### 分支 A — "单 Agent 怎么实现的?"
> 应对时长: 1-2 分钟, 回答完回到主线.
>
> "单 Agent 是 `ERPCopilotAgent` (`macs_pkg/erp/agents/copilot_agent.py:203-260`), 注册 7 个 tool — 5 个 MCP 库存销售采购工具, 1 个 RAG 检索, 1 个 NL→SQL. LLM 根据用户 query 自动选 tool. 我实际跑下来 prompt 会膨胀, 因为工具描述越来越长, 调用越来越慢, 所以后来才拆多 Agent. · 单 Agent 的优势是实现简单, 缺点是职责不分离, 一个 tool 挂了整体挂. 多 Agent 用 Hierarchical 编排, Planner 只管拆任务, Executor 只管执行."

### 分支 B — "你这个项目最大亮点是什么?"
> 应对时长: 1 分钟, 然后引导到具体技术深挖.
>
> "我觉得是**工程化完整度**. 168 测试 + CI + Web UI + 健康检查 + 4 层 SQL 防护, 加上 5 张表的真实数据, 这是 95% 候选人项目做不到的. Plan 没要求这些, 是我主动做的. · 具体我可以展开讲, 比如健康检查为什么做成单一事实源, 比如 SQL 防护为什么选 AST 解析而不是字符串过滤, 比如测试为什么按模块分 12 个文件. 您想听哪个?"

### 分支 C — "用了哪些 LLM? 为什么?"
> 应对时长: 1 分钟.
>
> "7 个 Provider: MiniMax、Claude、Qwen、Zhipu、DeepSeek、Hunyuan、OpenAI. 我抽象成统一接口, 切换 Provider 只改 2 处 — `build_default_provider()` 工厂函数和环境变量. 这样在国内 (Qwen/DeepSeek/Zhipu) 和海外 (Claude/OpenAI) 都能跑, 应对不同客户的合规要求. · 实际业务跑我用 Claude Sonnet 4 和 Qwen Plus, 因为这两个在中文 ERP 场景下效果和成本平衡最好."

## 七、3 个常见雷区

> 准备时规避, 面试时一旦发现苗头立即收.

### 雷区 1 — 把项目说成"另一个 LangChain"
**症状**: "我做了个多 Agent 框架, 支持工具调用, 支持 RAG, 支持..."
**修正**: 改口 "我做了一个**面向 ERP 业务场景的 AI 助手产品**", 第一句话就强调业务场景, 不是技术框架.

### 雷区 2 — 技术细节讲得比业务多
**症状**: 5 分钟里 4 分钟讲 AST 解析、async pool、psycopg 绑定, 1 分钟讲业务.
**修正**: 业务 (库存风险 / 采购建议 / 制度问答) 和技术 (4 层防护 / 多 Agent) 至少 1:1 配比. 听众是面试官, 不是技术 reviewer.

### 雷区 3 — 失败案例讲成"我队友的锅"
**症状**: "框架的 RuntimeEngine 设计有问题, 导致我没法做..."
**修正**: 失败案例的主体是"我". "我设计的接口丢了异常类型, 我反思后改成暴露 `error_type`" — 这是工程师心态, 不是甩锅心态.
